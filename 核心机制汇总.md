# 09 Hermes 核心机制汇总

> 本文汇总 Hermes Agent 五大核心机制：Agent 核心循环、上下文压缩、Memory 系统（含 Flush）、沙箱安全、Gateway 会话管理。适合快速建立对 Hermes 技术架构的整体认知。

## 一、Agent 核心循环

### 1.1 架构概览

核心引擎是 `run_agent.py` 中的 **AIAgent** 类（~10,500 行），采用**同步 while 循环 + 回调**的架构风格。

![AIAgent 生命周期——构造→初始化→循环→清理，四阶段流转](./imgs/02-agent-lifecycle.png)

```
用户输入 → 构建 system prompt → 调 LLM API
                                    ↓
                              模型返回响应
                                    ↓
                        ┌─── tool_use? ───┐
                        ↓ Yes             ↓ No
                   权限检查            输出结果
                        ↓              结束循环
                   执行工具
                        ↓
                   结果追加到消息历史
                        ↓
                   压缩检查 → 回到调 API
```

### 1.2 单轮生命周期

![Agent 主循环环形图——build prompt → call API → parse response → execute tools → compress → continue](./imgs/02-main-loop.png)

1. 生成 `task_id`
2. 将用户消息追加到对话历史
3. 通过 `prompt_builder.py` 构建或复用缓存的系统提示
4. 检查预压缩（上下文超过窗口 50% 时触发）
5. 构建 API 消息，按模式适配格式
6. 注入临时提示层（预算警告、上下文压力提示）
7. 应用 Anthropic prompt caching
8. 发起可中断的流式 API 调用
9. 解析响应：若有 tool_calls → 执行工具 → 循环；若为文本响应 → 持久化 → 返回

![AIAgent 回调系统——中心 AIAgent，周围辐射 5 条线连接 tool_progress/tool_start/step/reasoning/completion 回调，Agent 的神经末梢](./imgs/02-callback-system.png)

### 1.3 三种 API 模式

| 模式 | 用途 | 客户端 |
|------|------|--------|
| `chat_completions` | OpenAI 兼容端点 | `openai.OpenAI` |
| `codex_responses` | OpenAI Codex / Responses API | `openai.OpenAI` |
| `anthropic_messages` | 原生 Anthropic Messages API | `anthropic.Anthropic` |

三种模式收敛到相同的内部消息格式（OpenAI 风格的 role/content/tool_calls 字典）。

### 1.4 IterationBudget：油量表机制

```python
class IterationBudget:
    max_total: int        # 默认 90
    _used: int            # 线程安全计数器
    consume() -> bool     # 消费一次迭代
    refund() -> None      # 轻量级 RPC 调用免费
```

两级预算预警系统：
- **70%**：轻推 —— "开始收尾"
- **90%**：最后通牒 —— "立刻给出最终回答"

警告注入到最后一条工具结果中，旧警告每轮清除，防止模型在后续 turn 里持续受干扰。

![IterationBudget 预算仪表盘——90/90 剩余迭代，指针从绿到黄到红](./imgs/02-iteration-budget.png)

### 1.5 并行工具执行

**三阶段安全检查：**

1. **黑名单拦截**：`clarify` 等交互式工具，出现在批次中则整批降级为串行
2. **白名单放行**：`read_file`、`web_search` 等只读工具无条件并行
3. **路径冲突检测**：`read_file`、`write_file`、`patch` 只有目标路径不重叠时才能并行

```python
_NEVER_PARALLEL_TOOLS = frozenset({"clarify"})
_PARALLEL_SAFE_TOOLS = frozenset({
    "read_file", "search_files", "session_search",
    "web_extract", "web_search", "vision_analyze", ...
})
_MAX_TOOL_WORKERS = 8
```

通过安全检查后走 `ThreadPoolExecutor`，最多 8 个 worker，结果按原始顺序收集。

![ThreadPoolExecutor 并行执行——分发节点 → 4 条并行线 → 工具执行 → 汇聚回结果收集点](./imgs/02-parallel-tools.png)

### 1.6 工具结果三级防御体系

防止单个工具结果撑爆上下文窗口：

| 层级 | 机制 | 说明 |
|------|------|------|
| **第 1 级** | 工具自限 | `search_files` 等工具预先截断输出 |
| **第 2 级** | 单结果落盘 | 超阈值的写入 `/tmp/hermes-results/`，替换为 preview + 文件路径 |
| **第 3 级** | 整轮预算 | 本轮所有结果 > 200K 字符，最大的强制落盘 |

![工具结果三级处理——小结果直接返回，大结果落盘 /tmp/hermes-results/，阈值线分界](./imgs/02-tool-result-persist.png)

### 1.7 消息清洗

每轮 API 调用前的关键清洗操作：

| 操作 | 触发场景 | 处理方式 |
|------|----------|----------|
| Surrogate 消毒 | 富文本粘贴 | 替换为 U+FFFD |
| Budget Warning 剥离 | 每轮开头 | JSON del / 正则移除 |
| 孤儿 tool_call 修复 | 压缩/手动编辑后 | 删孤儿 result / 补 stub |
| reasoning 字段转换 | 多轮推理上下文 | reasoning → reasoning_content |

![消息清洗流水线——脏消息经过 3 个过滤器（budget warning/empty content/malformed），干净消息输出](./imgs/02-message-sanitize.png)

### 1.8 错误重试与 Fallback

**Jittered Backoff**：用 `time_ns ^ (tick * 0x9E3779B9)` 做独立种子，避免多个 Gateway 会话同步重试产生雷同抖动值。

**Fallback 链**：主 provider → fallback[0] → fallback[1] → ... 逐级降级。每个新 turn 恢复主 provider。

**关键错误处理**：

| 错误类型 | 处理 |
|----------|------|
| 429 Rate Limit | jittered_backoff → fallback |
| 上下文超限 | 压缩 → 降级上下文窗口探测 |
| 工具名幻觉 | 自动修复 → 注入错误消息让模型自纠 |
| 流式连接僵死 | 90s stale-stream 检测 + 60s 读超时 |

![重试阶梯——3 层台阶逐级升高（retry 1/2/3），最后一层 fallback 箭头指向备选 provider](./imgs/02-retry-fallback.png)

### 1.9 始终流式 (Always Streaming)

Hermes 始终使用 streaming 路径，即使没有流式消费者。非流式调用存在隐蔽问题：provider 用 SSE keep-alive ping 保持连接但不返回数据，Agent 无限挂住。流式路径自带 90 秒健康检查和 60 秒读超时。

![流式处理——API token 流像水管流向终端，中间有 buffer 和 flush 节点](./imgs/02-streaming.png)

### 1.10 与 Claude Code 循环架构对比

| 维度 | Hermes Agent (Python) | Claude Code (TypeScript) |
|------|----------------------|--------------------------|
| 核心循环 | 同步 `while` + callback | `async function*` yield |
| 并发模型 | `ThreadPoolExecutor` | 原生 `Promise.all` |
| 中断机制 | `_interrupt_requested` 标志位 | AbortController signal |
| 流式传播 | 8 个 callback 插槽 | yield 到上层 for-await-of |
| 文件组织 | 10,500 行单文件 | 分散在多个模块 |

![同步 while vs AsyncGenerator——左为 Python while loop 方框循环，右为 JS yield 暂停恢复箭头，中间 VS 分隔](./imgs/02-sync-vs-async.png)

## 二、上下文压缩

### 2.1 压缩触发策略

| 触发条件 | 阈值 | 时机 |
|----------|------|------|
| **预压缩** | 上下文超过窗口 50% | API 调用前 |
| **网关自动压缩** | 上下文超过 85% | 轮次之间 |

50% 触发看似激进，但配合迭代摘要策略（早压缩、多次压缩、每次增量更新），单次被压缩的内容量更小，摘要质量反而更高。

![上下文压缩全景——左侧 200K tokens 消息列表即将溢出，中间漏斗压缩器（context_compressor.py），右侧精简输出 HEAD + SUMMARY + TAIL](./imgs/05-compression-overview.png)

![两阶段压缩——Phase 1: 剪刀裁剪 >200 字符的 tool result 为占位符（零成本）；Phase 2: LLM 将多条消息摘要为一条结构化摘要（预算=内容×20%）](./imgs/05-two-phase-compression.png)

![阈值预算水位——200K 上下文窗口柱状图，50% 黄色虚线触发线 = 100K，标注 tail_token_budget=20K、max_summary_tokens=10K](./imgs/05-threshold-budget.png)

```
Step 0: Memory Flush（防止压缩丢失信息）
    ↓
Step 1: 裁剪工具输出（>200 字符的旧 tool result → 占位符，零成本）
    ↓
Step 2: 确定压缩边界
    head_end = protect_first_n (3 条，对齐到非 tool 消息)
    tail_start = _find_tail_cut_by_tokens (≥20 条 或 ≥20K tokens)
    ↓
Step 3: LLM 结构化摘要
    首次：从零生成七段式结构化摘要
    后续：在 _previous_summary 基础上增量更新
    预算：max(2000, min(内容量×20%, 窗口×5%))
    ↓
Step 4: 组装压缩后消息
    HEAD (3条) + SUMMARY (1条) + TAIL (~20K tokens)
    _sanitize_tool_pairs 修复孤儿配对
    ↓
Step 5: Session 分裂
    旧 session → end_reason="compression"
    新 session → parent_session_id = 旧 session
```

### 2.3 头尾保护策略

- **头部保护**：固定 3 条消息（system prompt、第一条用户消息、第一条 assistant 回复）
- **尾部保护**：Token 预算制（默认 20K tokens），同时保留 `protect_last_n=20` 作为保底下限

这种双重保护兼顾两种情况：短消息场景能保留更多上下文，长 tool result 场景能兜底保护最近 20 条。

![头尾保护策略——HEAD 区盾牌标注"保护区 3 条"（system/user[0]/asst[0]），MIDDLE 虚线框"可压缩区"，TAIL 盾牌"保护区 ~20K tokens / ≥20 条"](./imgs/05-head-tail-protection.png)

![Token 预算制 vs 固定消息数——左侧固定 20 条框住参差不齐的方块，右侧 20K tokens 虚线精准划定边界 + ≥20 条保底线](./imgs/05-token-budget-vs-fixed.png)

![七段式结构化摘要模板卡片——左侧纵向 7 个 section：Goal、Constraints、Progress（含 Done/In Progress/Blocked）、Key Decisions、Relevant Files、Next Steps、Critical Context；右侧弯曲箭头展示迭代更新 v1→新内容→v2](./imgs/05-structured-summary.png)

```
## Goal
[用户在做什么]

## Constraints & Preferences
[用户偏好、编码风格、约束条件]

## Progress
### Done
[完成了什么 — 文件路径、执行命令、具体结果]
### In Progress
[正在做什么]
### Blocked
[卡在哪里]

## Key Decisions
[技术决策和理由]

## Relevant Files
[读过/改过/创建的文件列表]

## Next Steps
[下一步该做什么]

## Critical Context
[不能丢的具体值：错误信息、配置参数、数据]
```

结构化模板强制 LLM 按类别填充，避免自由格式摘要中关键信息被选择性丢弃。第二次压缩时不从零开始，而是在前一次摘要基础上增量更新。

### 2.5 Tool Call/Result 配对完整性

压缩不破坏消息格式的核心保障：

- `_sanitize_tool_pairs`：收集存活 tool_call ID → 删孤儿 result → 补缺失 stub
- `_align_boundary_forward`：起始边界跳过连续 tool result
- `_align_boundary_backward`：结束边界将 assistant + tool_results 整组拉进压缩区

### 2.6 Session 分裂链

```
Session_A (original)
  └─ end_reason: "compression"
      └─ Session_B (parent=A)
          └─ end_reason: "compression"
              └─ Session_C (parent=B) ← 当前活跃
```

三个好处：历史可追溯（session_search）、标题自动继承编号、成本归集可沿链聚合。

![链式 Session 分裂——Session A(原始) → compression → Session B(parent=A) → compression → Session C(parent=B) 当前活跃，时间线从左到右](./imgs/05-session-split-chain.png)

### 2.7 与 Claude Code / OpenClaw 压缩策略对比

| 维度 | Hermes | Claude Code | OpenClaw |
|------|--------|-------------|----------|
| 压缩方式 | LLM 结构化摘要 | 三层递进（微/中/全量）压缩 | 固定窗口裁剪 |
| 触发阈值 | 50% | 80%/90%/95% | 上下文接近上限 |
| 摘要迭代 | 增量更新 previous_summary | 每次从零生成 | 无 |
| Session 管理 | parent_session_id 链式 | 无 | 无 |
| 信息保留 | 摘要保留关键信息 | 分层保留 | 被裁剪的丢失 |
| 计算成本 | 每次 LLM 调用 | 多级 LLM 调用 | 零额外成本 |

![Claude Code vs Hermes 压缩对比——左侧 Claude Code 三层递进漏斗（80%/90%/95%），右侧 Hermes 两层漏斗（50% 单一阈值）+ 迭代更新/Session 分裂差异化优势](./imgs/05-vs-claude-code.png)

## 三、Memory 系统

![四层记忆架构——声明性记忆（MEMORY.md/USER.md） → 程序性记忆（Skill 自主创建） → 会话记忆（SQLite WAL + FTS5） → 训练记忆（batch_runner → compressor），左侧 MemoryManager 插件化架构标注，右侧自我进化闭环箭头](./imgs/07-memory-overview.png)

### 3.1 内置双文件记忆

| 文件 | 用途 | 字符限制 | 估计 Token |
|------|------|----------|-----------|
| **MEMORY.md** | Agent 个人笔记 — 环境事实、约定、经验 | 2,200 字符 | ~800 |
| **USER.md** | 用户档案 — 偏好、沟通风格、期望 | 1,375 字符 | ~500 |

**设计要点：**
- 会话开始时以**冻结快照**形式注入系统提示词，保证 prompt cache 稳定
- 写操作立即持久化到磁盘，但直到下次会话启动才出现在提示中
- `memory` 工具支持三种操作：`add`、`replace`（子字符串匹配）、`remove`
- 完全重复的条目静默拒绝
- 写入前扫描：提示注入、凭据泄露、SSH 后门、不可见 Unicode 字符

### 3.2 MemoryManager：插件化架构

```python
class MemoryManager:
    _providers: List[MemoryProvider]     # builtin 永远在第一位
    _tool_to_provider: Dict[str, ...]    # O(1) 路由
    _has_external: bool                  # 外部 provider 最多 1 个
```

**完整生命周期钩子：**

| 钩子 | 时机 | 用途 |
|------|------|------|
| `initialize` | Session 开始 | 连接后端、注入 hermes_home |
| `system_prompt_block` | System prompt 构建 | 静态记忆注入 |
| `on_turn_start` | 每轮开始 | 计数、scope 管理 |
| `prefetch` | API 调用前 | 基于当前 query 召回记忆 |
| `sync_turn` | 每轮完成后 | 异步写入对话记录 |
| `on_memory_write` | 内置记忆写入时 | 镜像到外部后端（跳过 builtin 自身）|
| `on_pre_compress` | 压缩前 | 从即将丢弃的消息中提取洞察 |
| `on_session_end` | 会话结束 | 端到端事实抽取 |
| `shutdown` | 关闭 | 反序关闭（后注册的先关）|

关键设计：每个钩子有 try-except 包裹，一个 provider 崩溃不拖垮另一个。`on_memory_write` 跳过 builtin 自身，防止写入循环。`on_pre_compress` 在压缩前抢救即将丢失的知识。

![MemoryProvider 生命周期时间线——initialize → system_prompt_block → Turn Loop（on_turn_start → prefetch → Model推理 → sync_turn → queue_prefetch）→ on_pre_compress（高亮：抢救即将丢弃的记忆）→ on_session_end → shutdown（反序关闭）](./imgs/07-lifecycle-hooks.png)

### 3.3 会话搜索 (session_search)

与持久化记忆互补的独立系统：

| 特性 | 持久化记忆 | 会话搜索 |
|------|-----------|---------|
| 容量 | ~1,300 tokens | 无限制 |
| 速度 | 即时（已在系统提示中）| ~20ms FTS5 |
| 成本 | 每次提示有 token 开销 | 免费（无 LLM 调用）|
| 使用场景 | 始终可用的关键事实 | 查找特定历史对话 |

基于 SQLite FTS5 全文搜索，直接从数据库返回原始消息，无需 LLM 摘要。

### 3.4 SQLite 持久化：WAL + 抖动重试

```python
class SessionDB:
    _WRITE_MAX_RETRIES = 15
    _WRITE_RETRY_MIN_S = 0.020    # 20ms
    _WRITE_RETRY_MAX_S = 0.150    # 150ms
    _CHECKPOINT_EVERY_N_WRITES = 50
```

- **WAL 模式**：多 reader + 单 writer 并发，读写不互斥
- **抖动重试**：20-150ms 随机抖动打破 convoy effect
- **BEGIN IMMEDIATE**：事务开始时抢写锁，冲突第一时间暴露
- **PASSIVE checkpoint**：每 50 次写入触发，不阻塞读取

![SQLite 存储架构——中心 state.db 圆柱（WAL 模式），左侧多条并发箭头（CLI 会话/Telegram/Discord/Worktree），右上抖动重试 20-150ms 打破护航效应，右下被动 Checkpoint PASSIVE 永不阻塞，底部 FTS5 全文搜索虚拟表 + 触发器自动同步](./imgs/07-sqlite-architecture.png)

### 3.6 四层记忆体系

1. **声明性记忆**（MEMORY.md / USER.md）：Agent 知道什么事实
2. **程序性记忆**（SKILL.md）：Agent 会做什么方法论
3. **会话记忆**（SQLite + FTS5）：Agent 做过什么历史
4. **训练记忆**（batch_runner + trajectory_compressor）：Agent 如何变得更好

四层协同构成从运行时能力到进化能力的完整闭环。

### 3.7 Memory Flush 机制

Memory Flush 是确保长期记忆在对话中持续积累的关键机制。它不是一次性操作，而是**多点触发、分层执行**的持续过程。

**触发时机（5 种）：**

| 触发条件 | 调用钩子 | 说明 |
|----------|---------|------|
| **上下文压缩前** | `on_pre_compress(messages)` | 从即将被丢弃的消息中提取洞察，返回文本注入压缩摘要 prompt |
| **会话结束时** | `on_session_end(messages)` | CLI 退出、`/reset` 命令、Gateway session 过期时触发 |
| **Session 切换时** | `on_session_switch` / `commit_memory_session()` | `/new` 命令、压缩导致 session 分裂时，先 flush 旧 session 再开始新的 |
| **每轮对话后** | `sync_turn(user, assistant)` | 异步写入对话记录到后端 |
| **周期性 Flush** | `flush_min_turns` 配置 | 会话退出/重置时，若 user turn 数 ≥ 阈值则触发 |

**压缩前 Flush 的完整流程：**

```
上下文超过 50% 阈值
        ↓
MemoryManager.on_pre_compress(messages)
        ↓
遍历所有 provider，调用各自的 on_pre_compress()
        ↓
┌─────────────────────────────────────────────────────┐
│ BuiltinProvider: 无操作（MEMORY.md 已通过           │
│ memory 工具实时持久化）                              │
├─────────────────────────────────────────────────────┤
│ Honcho: 从待压缩消息中提取用户偏好、约束、           │
│ 决策等结构化洞察                                    │
├─────────────────────────────────────────────────────┤
│ Holographic: 计算消息向量并写入本地存储             │
├─────────────────────────────────────────────────────┤
│ Hindsight: 更新知识图谱实体和关系                   │
└─────────────────────────────────────────────────────┘
        ↓
聚合所有 provider 返回的文本
        ↓
注入到压缩摘要 prompt（"以下是从即将丢弃的消息中
提取的关键信息，请保留到摘要中：..."）
        ↓
压缩器生成摘要（包含 provider 提取的洞察）
        ↓
中间消息被压缩，但知识不丢失
```

**关键设计原则：**

1. **压缩器压上下文，不压知识**：`on_pre_compress` 让 Memory provider 在消息被丢弃前抢救有价值信息，这些信息通过摘要形式保留在压缩后的上下文中。

2. **内置 vs 外部分离**：内置 MEMORY.md/USER.md 由 `memory` 工具实时写磁盘，不依赖 flush 钩子。外部 provider 依赖 `on_session_end` 和 `sync_turn` 做后端同步。

3. **防御式执行**：每个 provider 的 flush 钩子都有 try-except 包裹。一个 provider 的 flush 失败不会阻塞其他 provider，也不会阻塞压缩流程。

4. **Session 切换的原子性**：`commit_memory_session()` 先 flush 旧 session，再初始化新 session。Hindsight 等 provider 在 `on_session_switch` 时使用旧 session 标识符完成写入，避免数据归属错乱。

5. **Flush 索引重置**：上下文压缩创建新 session 时，flush 索引归零，确保同一批消息不会被重复提取。

## 四、沙箱安全机制

### 4.1 七层纵深防御

```
┌──────────────────────────────────────────────┐
│  第 1 层：危险命令审批                          │
│  20+ 模式匹配，三档策略（manual/smart/off）      │
│  硬阻止列表永不过滤                            │
├──────────────────────────────────────────────┤
│  第 2 层：用户授权（Gateway）                    │
│  分层检查 → DM 配对码（8 字符，1h 过期）         │
├──────────────────────────────────────────────┤
│  第 3 层：容器隔离                              │
│  Docker: --cap-drop ALL, no-new-privileges    │
│  资源限制: CPU/内存(5GB)/磁盘(50GB)             │
├──────────────────────────────────────────────┤
│  第 4 层：环境变量过滤                           │
│  剥离 KEY/TOKEN/SECRET/PASSWORD 等敏感变量      │
├──────────────────────────────────────────────┤
│  第 5 层：MCP 凭证处理 + SSRF 防护              │
│  工具结果中凭证模式替换为 [REDACTED]             │
│  SSRF: 阻止私有网络、回环、云元数据 IP             │
├──────────────────────────────────────────────┤
│  第 6 层：跨会话隔离 + 输入净化                   │
│  会话间数据不可访问，Cron 路径遍历保护            │
├──────────────────────────────────────────────┤
│  第 7 层：Tirith 预执行扫描                      │
│  同形异义词 URL 检测、管道注入检测               │
└──────────────────────────────────────────────┘
```

### 4.2 第 1 层：危险命令审批

**三种模式：**

| 模式 | 行为 |
|------|------|
| `manual`（默认）| 始终提示用户确认 |
| `smart` | 辅助 LLM 评估风险：低风险自动批准，明确危险自动拒绝，不确定的升级 |
| `off` | 禁用所有检查（等同 `--yolo`）|

**硬阻止列表（始终强制执行）：**
- `rm -rf /`
- 分叉炸弹
- 针对已挂载根设备的 `mkfs.*`
- `dd if=/dev/zero of=/dev/sd*`
- 在文件系统根目录将不信任 URL 通过管道传给 `sh`

**容器绕过：** 在 Docker、Singularity、Modal、Daytona 后端运行时，危险命令检查被跳过（容器本身就是安全边界）。

### 4.3 第 2 层：Gateway 用户授权

- **DM 配对码系统**：8 字符加密随机码（32 字符无歧义字母表，排除 0/O/1/I）
- 代码 1 小时后过期，速率限制：每用户每 10 分钟 1 次请求
- 5 次失败尝试后锁定 1 小时
- 配对数据文件权限 `chmod 0600`
- 代码永不记录到标准输出

### 4.4 第 3 层：容器隔离

```bash
# Docker 安全强化
--cap-drop ALL
--cap-add DAC_OVERRIDE,CHOWN,FOWNER
--security-opt no-new-privileges
--pids-limit 256
# tmpfs 挂载（noexec/nosuid）
```

可配置资源限制：CPU 核心数、内存（默认 5GB）、磁盘（默认 50GB）。

### 4.5 第 4 层：环境变量过滤

`execute_code` 和 `terminal` 从子进程中剥离敏感环境变量：
- 技能范围：SKILL.md 中声明的变量自动注册
- 基于配置：添加到 `terminal.env_passthrough`
- 过滤规则：阻止包含 KEY/TOKEN/SECRET/PASSWORD/CREDENTIAL/PASSWD/AUTH 的变量
- Docker 默认不传递任何环境变量

### 4.6 第 5 层：MCP 凭证 + SSRF

**凭证脱敏**：工具错误消息中的 `ghp_...`、`sk-...`、bearer token、`token=`、`key=`、`API_KEY=`、`password=`、`secret=` 等模式替换为 `[REDACTED]`。

**SSRF 防护**：
- 阻止私有网络（RFC 1918）、回环地址
- 阻止链路本地地址（含 `169.254.169.254`）
- 阻止 CGNAT（`100.64.0.0/10`）
- DNS 失败视为阻止（失败关闭）
- 重定向链每一跳重新验证
- 可配置白名单：`security.allow_private_urls: true`

### 4.7 第 6-7 层：会话隔离 + Tirith 扫描

**跨会话隔离**：会话无法访问彼此数据，Cron 任务路径经过路径遍历保护。

**上下文文件注入保护**：AGENTS.md、.cursorrules、SOUL.md 加载前扫描：
- 忽略/覆盖先前指令的指示
- 隐藏 HTML 注释中的可疑关键词
- 读取机密文件的尝试
- 通过 curl 泄露凭据
- 不可见 Unicode 字符

**Tirith 集成**：同形异义词 URL 欺骗检测、管道到解释器模式检测、终端注入检测。

### 4.8 终端后端安全对比

| 后端 | 隔离级别 | 危险命令检查 | 适用场景 |
|------|---------|-------------|---------|
| local | 无（主机）| 是 | 开发、可信用户 |
| ssh | 远程机器 | 是 | 分离服务器 |
| docker | 容器 | 跳过 | 生产 Gateway |
| singularity | 容器 | 跳过 | HPC 环境 |
| modal | 云沙箱 | 跳过 | 可扩展云隔离 |
| daytona | 云沙箱 | 跳过 | 持久云工作区 |

### 4.9 生产部署清单

1. 设置显式 allowlist（生产环境禁 `GATEWAY_ALLOW_ALL_USERS=true`）
2. 使用容器后端（`terminal.backend: docker`）
3. 限制资源上限（CPU、内存、磁盘）
4. 安全存储密钥（`~/.hermes/.env`，正确文件权限）
5. 启用 DM 配对
6. 审计命令允许列表
7. 设置 `MESSAGING_CWD` 防止在敏感目录操作
8. 以非 root 用户运行 Gateway
9. 监控日志 `~/.hermes/logs/`
10. 定期 `hermes update`

## 五、Gateway 会话管理

![Gateway 全景——中心 GatewayRunner 枢纽（7,620 行·单进程多适配器），上方半圆排列 8 个平台图标（Telegram/Discord/Slack/WeChat/Feishu/Email/Signal/WhatsApp），左侧 SessionStore（per-user per-platform），右侧 DeliveryRouter（跨平台投递），下方 AIAgent 统一消息处理](./imgs/06-gateway-overview.png)

### 5.1 会话生命周期

Gateway 模式下，每个聊天平台的用户对话被抽象为 **Session**，由 `SessionStore` 统一管理。每个会话分配唯一 `session_id`（格式 `YYYYMMDD_HHMMSS_<hex>`），通过 `SessionEntry` 追踪元数据。

```
会话创建 → 活跃对话 → [空闲] → 过期检测 → 最终化 → 清理
                ↑                                    |
                └──── 用户恢复会话 ←──────────────────┘
                           (新 session_id)
```

### 5.2 会话过期策略 (Session Reset Policies)

四种可配置策略，通过 `GatewayConfig` 的 `session_reset_policy` 设置，可按平台、会话类型分别覆盖：

| 策略 | 行为 | 配置参数 |
|------|------|---------|
| **`idle`** | 超过指定分钟数无活动后过期 | `idle_minutes`（默认 1440，即 24 小时）|
| **`daily`** | 每天在指定小时过期 | `at_hour`（如 `3` = 凌晨 3 点）|
| **`both`** | 任一条件先满足即过期 | `idle_minutes` + `at_hour` |
| **`none`** | 永不过期 | 无 |

**特殊保护：** 有活跃后台进程的会话永远不会被视为过期，即使满足过期条件。

过期判定由 `SessionStore._is_session_expired()` 执行，以 `SessionEntry.updated_at` 作为活动时间戳。

![Session 生命周期流——顶部"用户消息到达" → 决策菱形"会话存在？" → 三条路径分支（复用已有会话 / 记忆刷写→新建会话 / 直接新建），汇聚到活跃会话；右侧后台 Watcher "每 5 分钟扫描 → 检查过期 → flush 记忆"循环](./imgs/06-session-store.png)

### 5.3 _session_expiry_watcher：后台过期监控

`GatewayRunner` 内部运行一个异步后台任务，周期性扫描并处理过期会话：

```python
# 伪代码示意
async def _session_expiry_watcher(self, interval: int = 300):
    while self._running:
        await asyncio.sleep(interval)  # 默认 300 秒（5 分钟）
        for entry in self._session_store.get_expired_entries():
            await self._finalize_expired_session(entry)
```

**最终化流程（_finalize_expired_session）：**

```
1. on_session_finalize hooks
   → 触发插件清理、保存最终 session 指标
   → Memory provider 执行 on_session_end()，flush 缓冲消息
        ↓
2. _cleanup_agent_resources(cached_agent)
   → 调用 AIAgent.close()
   → 拆除 terminal sandbox、释放 LLM clients
   → 清理 tool state（"硬"清理）
        ↓
3. _evict_cached_agent(session_key)
   → 从 agent_cache 中移除 AIAgent 实例
   → 允许 GC 回收（释放内存）
        ↓
4. 标记 expiry_finalized = True
   → 持久化到 sessions.json
   → 防止 Gateway 重启后重复最终化
```

**失败重试：** 若最终化失败，最多重试 `_MAX_FINALIZE_RETRIES`（3 次），之后强制标记为已最终化，防止无限重试循环。

### 5.4 Agent 缓存空闲淘汰

除了策略驱动的会话过期，Gateway 还有一层 **Agent 实例缓存管理**：

| 机制 | 触发条件 | 操作 |
|------|---------|------|
| **LRU 空闲淘汰** | Agent 空闲超过 `_AGENT_CACHE_IDLE_TTL_SECS`（3600s = 1 小时）| `release_clients()` 释放 LLM 连接（"软"清理），**保留 tool state**（terminal sandbox 不销毁），允许后续 resume 复用 `task_id` |
| **Stale 运行中 Agent 淘汰** | Agent 运行中但空闲超过 `HERMES_AGENT_TIMEOUT`（默认 1800s）或 wall-clock 超限（10x timeout 或 2h，取大值）| 作废 run generation，强制释放状态 |

`_AGENT_PENDING_SENTINEL` 是一个特殊哨兵值，在异步 agent 初始化期间占位，stale agent 淘汰时显式跳过，防止误清理正在创建的 agent。

### 5.5 会话过期时的状态处理

| 状态类别 | 处理方式 |
|----------|---------|
| **Agent 实例** | 从缓存驱逐，GC 回收，内存状态丢弃 |
| **Tool 资源** | `AIAgent.close()` 硬拆除 terminal sandbox 等资源 |
| **LLM 客户端** | `release_clients()` 释放连接 |
| **Memory** | 过期前给 agent 一个 turn 保存重要记忆和技能；`on_session_finalize` 触发 provider flush |
| **Session 元数据** | `expiry_finalized = True` 持久化，防止重复处理 |

### 5.6 SessionStore 清理与裁剪

`SessionStore` 定期清理过期的 `SessionEntry` 记录，防止内存和 `sessions.json` 无限增长：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `session_store_max_age_days` | 90 天 | 已结束 session 的最大保留天数 |
| `min_interval_hours` | 24 小时 | 自动裁剪的最小间隔 |

裁剪是**惰性删除**：若被裁剪的 session 后续被用户恢复，会获得全新 `session_id`，效果等同于 reset policy 触发。默认禁用自动裁剪，需在 `config.yaml` 中显式启用。

### 5.7 关键 TTL/超时汇总

| 参数 | 默认值 | 作用域 |
|------|--------|--------|
| `idle_minutes` | 1440（24h）| Session 空闲过期 |
| `_session_expiry_watcher interval` | 300s（5min）| 过期扫描频率 |
| `_AGENT_CACHE_IDLE_TTL_SECS` | 3600s（1h）| Agent 缓存软淘汰 |
| `HERMES_AGENT_TIMEOUT` | 1800s（30min）| Stale 运行中 Agent 检测 |
| `session_store_max_age_days` | 90 天 | Session 记录保留期 |
| `min_interval_hours` | 24h | 自动裁剪间隔 |
| `_MAX_FINALIZE_RETRIES` | 3 | 最终化失败重试次数 |

### 5.8 显式 Session 重置

用户主动触发的重置（`/new`、`/reset` 命令）：

```
旧 session:
  → on_session_finalize hook 触发
  → on_session_reset hook 触发
  → AIAgent 状态清零:
     - session_id 更新
     - conversation_history 清空
     - _last_flushed_db_idx 重置
     - _todo_store 清空
  → MemoryManager.on_session_switch(reset=True)
     → provider flush 累积的 per-session 状态
新 session:
  → 全新 session_id
  → 干净的对话上下文
```
